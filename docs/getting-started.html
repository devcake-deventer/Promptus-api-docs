Getting Started

Welcome to the Promptus API documentation. In these documentation pages, we'll get you up to speed with the usage of our API. Using our API you'll be able to integrate various Promptus features into your own website or application, including user registration, creating and managing courses, and creating invoices.

Before we get started...

GraphQL

Since our API follows the GraphQL specifications, we assume you have basic knowledge of consuming GraphQL APIs. GraphQL makes querying and mutating data very easy. It also enables you to specify which data you want to receive. Libraries for GraphQL exist in every major language to make this even easier, such as C#/.NET, JavaScript, and numerous others.

Note: Some API calls - such as logging users in and resetting passwords - do not use GraphQL. These API calls will be specified on these pages.

Promptus license

Additionally, we assume your company is a Promptus customer with a valid license to use our API. If this is not the case (yet), please contact Promptus to become a customer!

Using provided details

When your company has subscribed to become a Promptus client, an initial administrator account will be created for you. You'll also receive an email with details to connect to your own secure environment within Promptus. The email contains the following information, which is required to communicate with our API:

  • Subdomain
    Your company has a unique subdomain, which should be used in all API calls to specify your company's secured environment:
https://{subdomain}.api.lll.academy/graphql
  • Promptus application token
    The Promptus application token should be included in the headers of every request. It is used to identify and validate your company as a valid Promptus client:
  "Content-Type": "application/json",
  "X-Promptus-Authentication": "$APP_TOKEN"

Note: Not providing the application/json may result in unexpected results when a non-valid token is provided

When you fail to provide a valid token an error will be returned in the form of a JSON response. The request will fail with status code 412 and the response matches the one below.

{
    "message": "A non-valid app token is provided. Please make sure you're providing a valid one"
}

Your First Query

So how does our GraphQL API work, exactly? To illustrate that, we will build a simple query to get an overview of all available courses. To achieve this, the body of your request should look like this:

{
    courses {
        id
        code
        description
    }
}

Remember to include the 'X-Promptus-Authentication' and 'Content-Type' header, that we specified earlier. Because this is GraphQL, you should always use the POST method in your request. The endpoint is, per GraphQL specification, 'graphql', making the complete URL:

https://{subdomain}.api.lll.academy/graphql

Note: We advise using a tool like Insomnia to play around with our API. Insomnia's editor also provides auto-complete features and more detailed, field-specific documentation.

Depending on the contents of your database, the response should look something like this:

{
  "data": {
    "courses": [
      {
        "id": 2,
        "code": "CS-101",
        "description": "Introduction to computer science" 
      },
      {
        "id": 3,
        "code": "CS-102",
        "description": "The basics of computer science" 
      }
    ]
  }
}

This query retrieves the id, code, and description of the current courses. But what if we don't need the code, and instead want to know the price and how many places are still available? We can simply change the query like so:

{
    courses {
        id
-       code
        description
+       price
+       available_places
    }
}

Your response should now change accordingly:

{
  "data": {
    "courses": [
      {
        "id": 2,
        "description": "Introduction to computer science",
        "price": "100",
        "available_places": 22
      },
      {
        "id": 3,
        "description": "The basics of computer science",
        "price": "200",
        "available_places": 18
      }
    ]
  }
}

Mutations

When using GraphQL requests that create, update or delete data on the are called mutations. An example of a simple mutation is shown below. Depending on the query or mutation, you may need to provide one or more parameters.

mutation {
    mutation_name(param: value, param2: value) {
        field1,
        field2
    }
}

Errors

Due to the inner workings of GraphQl, when an error has been encountered the status code still is 200. This makes it especially important to provide consistent error handling. Fortunatily, when an error has been encountered the response always contains an error array with the errors encountered. For example, the mutation shown below is missing a required parameter.

mutation {
  registerUser(input: {first_name: "John", email: "john-doe@test.nl", password: "secret"}) {
    id
    first_name
  }
}

This will result in the following error to be thrown.

{
  "errors": [
    {
      "message": "Field RegisterUserInput.surname of required type String! was not provided.",
      "extensions": {
        "category": "graphql"
      },
      "locations": [
        {
          "line": 2,
          "column": 23
        }
      ]
    }
  ]
}

Loading Relations

It is also possible to load related objects in a single query. For example, you might want to show accreditations or study points.

{
    courses {
        id
        description
+       studyPoints { 
+           points 
+           description
+       }
+       accreditations {
+           name
+       }
    }
}

An object can even have references to its own type. For example, courses can have a parent-course and multiple sub-courses. When you request relations, you can specify the fields, similar to a normal query.

{
    courses {
        id
        description
+       parentCourse {
+           code
+       }
+       subCourses {
+           subject
+       }
    }
}

Course

Courses are another key component of the Promptus Api. Courses can be offered as seperate modules but they can also be part of a study. More information about this in insert link to that part of the documentation.

Retrieving courses

Courses can be retrieved from the Promptus API in two manners, either paginated or as a single list. The first is the simplest, but might not always be the best choice. A simple query to get the list of courses and some fields might look as follows.

{
    courses{
        id,
        description
    }
}

By default, the query will exclude sub-courses, template courses, cancelled and archived courses, and those which have no places remaining or are closed to enrollments. Using parameters, you can change this behaviour.

{
-   courses{
+   courses(check_places: false, include_templates: true){
        id,
        description
    }
}

The parameters you can use here are:

  • parent_course: If an ID is provided, will only show sub-courses of that course. If null is provided, this will show courses regardless of their parent course.
  • include_templates: If true, this will also show template courses.
  • not_archived: if false, this will include all courses that have been archived.
  • check_places: if false, this will include courses with zero places remaining.
  • show_cancelled: if true, also shows courses that have been cancelled
  • uses_dates: if true, checks if the current time is between the registration_start_date and registration_end_date, omitting this value will default it to true

Obtaining registrations of courses - Administrators only

In addition to all the fields that can be queried on the course model, administrators can also query all users that registered for this course. An example is shown below.

{
    courses{
        id,
        description
+       courseRegistrations {
+           id
+       }
    }
}

Administrators can also filter on certain column(s). Currently we support course registration filtering by a single field: confirmation_date. We can apply the filter using the following method:

{
-  courses {    
+  courses(hasCourseRegistrations: {column: CONFIRMATION_DATE, operator: EQ, value: null}) {
    id
    description
    courseRegistrations {
      id
      confirmation_date
    }
  }
}

Doing this allows administrators to filter courses by the confirmation date of their related course registrations. All basic SQL operators are supported but the most common ones are:

  • EQ
  • NEQ

Get course by id

There are several reasons as to why you might want to retrieve a specific course instead of a list of courses. An example of this might be that a user sees a list of courses and selects one to get more details. An example on how to perform such a query is shown below.

{
  courseByID(id: 1) {
    id
    description
  }
}

Check the API documentation to see all possible values that can be queried.

Registering for a course

A user can be registered for a course using the registerForCourse mutation. This requires a single argument for the course.

mutation{
    registerForCourse(course: 42){
        id
    }
}

By default, the user will be loaded from the authentication context, but this can be overriden in two ways. First, we can explicitly define the user to register:

mutation{
-   registerForCourse(course: 42){
+   registerForCourse(course: 42, user: 13){
        id
    }
}

Or, we can define a name which will create a user object for you.

mutation{
-   registerForCourse(course: 42){
+   registerForCourse(course: 42, first_name: "Steve", surname: "Wozniak"){
        id
    }
}

There are also other, optional parameters:

  • study_registration: Use this if the registration for this course is part of a registration for a study.
  • study: Which study is the registration part of.
  • intermediary_user: Is a person other than the student managing this registration? This might be an HR manager or someone else in the organisation.

Create a course - Administrators only

As an administrator you are able to create new courses that can be fetched by client applications. To provide this functionality the createCourse mutation is created.

mutation {
  createCourse(subject: "subject", description: "description")  {
    id
    subject
    description
  }
}

Required parameters

The createCourse mutation only has two required parameters: subject & description (1). Although we advise to fill in as much of the information as possible. This will help users of your application to make an informed decision about a course.

Optional parameters

Update a course - Administrators only

Administrators should be able to update a course for a number of different reasons. An example of such a reason is to update the price for a course. We offer this functionality through the updateCourse mutation.

mutation {
  updateCourse(id: 1, description: "updated description") {
    id
    archived_date
  }
}

Required parameters

This mutation has a single required parameter and this is the id of the course. When you fail to provide the id, or the id could not be resolved to a valid course object an exception will be thrown.

Optional parameters

Using this mutation a number of different parameters can be updated. In the table below we share more details and their meaning.

FieldTypeDescription
parent_courseIDAn unique ID pointing to a different course that is the parent of this course
pricePricePrice of the course
periodStringA human readable description of when the course takes place e.g. 'summer of 2021'
commentStringField to store a comment about this course
subjectStringDetails the subject of the course
descriptionStringA complete description about the course that can be shown to users
min_participantsIntThe minimal amount of participants required for this course to take place
max_participantsIntThe maxmimum amount of participants for a course
course_numberStringA number to identify the course
project_codeStringA code to identify the project
head_teacher_idIDSets the relationship between the user (head teacher) and the course
btw_idIDSets the relationship between a BTW percentage and the course
course_typeIDSets the relationship between a course type and the course
contentStringStore information about the content of the course
learning_goalsStringList of learning goals for the course
templateBooleanIndicates that the course is a template and users shouldn't sign up for this.
archivedBooleanDetermines if the course is archived (1)

certificate_validity_startDateTimeIndicates the moment that the certificate will become valid

Delete a course - Administrators only

Sometimes a course is no longer needed. In these cases we enable administrators to delete courses using one of two mutations: deleteCourse & forceDeleteCourse

Difference between the two mutations

We offer two ways of deleting courses. The deleteCourse mutation uses a principle called soft deletion. This means that the soft deleted courses are no longer included in queries but they still remain in the database. This makes it possible to restore courses that are deleted by accident. This is done using the restoreCourse mutation.

In contrast to this we also offer a solution for cases where a course should actually be deleted from the database. For those use cases we offer the forceDeleteCourse mutation.

Delete course

As described in the previous chapter this mutation makes use of a concept called soft deletion. This does not actually deletes the course from the database but simply makes sure that soft deleted courses are excluded from all queries.

mutation {
  deleteCourse(id: 1) {
    id
    code
    description
    deleted_at
  }
}

Restore course

Sometimes a course has been deleted by accident or you might need to restore it for another reason. When it was deleted using the soft delete concept this is done using the restoreCourse mutation.

mutation {
  restoreCourse(id: 1) {
    id
    code
    description
    deleted_at
  }
}

Force delete course

When simply excluding a course from queries isn't enough this mutation can be used to completely remove the course from the database.

mutation {
  forceDeleteCourse(id: 1) {
    id
    code
    description
    deleted_at
  }
}

Courses

Courses are another key component of the Promptus API. A course generally includes a set of course sessions that take place over a period of time. A course object therefore holds abstract information, such as subject and price, but not the information of a single session (e.g. time and location). Instead, this information is stored inside course_session object.

Course participants

When a user want to participate in a course, he or she should register for a course and NOT a course session. This may feel counter-intuitive. However, by design, registering for a course means the user will be a participant in all course sessions belonging to the course. This is doen, since participating in a course generally means you have to be present in all sessions. Therefore, a user does not have to register for individual course sessions.

Courses hierarchy

Courses can be placed within a hierarchy through the parentCourse and subCourses relations. This means that a single course can have multiple sub courses.This can be helpfull when you want to group together courses. For example, you might want to offer the same course multiple times in a single year. You can create a parant course (e.g. "Computer Science 2022") with several sub courses (e.g. CS - Q1 2022 and CS - Q3 2022).
By default, the Promptus mobile dashboard displays sibling courses (subcourses of the same parent) in a collapsable group.

Get course by id

There are several reasons as to why you might want to retrieve a specific course instead of a list of courses. An example of this might be that a user sees a list of courses and selects one to get more details. An example on how to perform such a query is shown below.

{
  course(id: 1) {
    id
    subject
  }
}

Return value

This returns the course with the corresponding id.

{
  "data": {
    "course": {
      "id": 1,
      "subject": "Computer Science"
    }
  }
}

Authorization

None.

Get a list of all courses

You can get a complete list of courses being offered using the following query:

{
    courses() {
        id,
        subject
    }
}

By default, the query will exclude sub-courses, template courses, cancelled and archived courses, and those which have no places remaining or are closed to enrollments. Using parameters, you can change this behaviour.

{
-   courses{
+   courses(check_places: false, include_templates: true){
        id,
        description
    }
}

The parameters you can use here are:

  • parent_course: If an ID is provided, will only show sub-courses of that course. If null is provided, this will show courses regardless of their parent course.
  • include_templates: If true, this will also show template courses.
  • not_archived: if false, this will include all courses that have been archived.
  • check_places: if false, this will include courses with zero places remaining.
  • show_cancelled: if true, also shows courses that have been cancelled
  • uses_dates: if true, checks if the current time is between the registration_start_date and registration_end_date, omitting this value will default it to true

Return value

This returns a list of courses, filtered by the given parameters.

{
  "data": {
    "courses": [
      {
        "id": "2",
        "subject": "Praktisch leidinggeven"
      },
      {
        "id": "4",
        "subject": "Online leren"
      },
      {
        "id": "5",
        "subject": "3D printen voor de maakindustrie"
      }
    ]
  }
}

Authorization

None.

Obtaining registrations of courses

In addition to all the fields that can be queried on the course model, admins can also query all registrations of a course. Following the graphQL principle, for each registration we can query the user. An example is shown below.

{
  courses(check_places: false uses_dates: false) {
    id
    subject
    courseRegistrations {
      id
      price
      user {
        first_name
      }
    }
  }
}

Return value

This returns the active user, based on provided bearer token.

Querying new registrations

We can even filter on columns of course registration. Currently we support course registration filtering by a single field: confirmation_date. This can be used to identify new and/or unconfirmed registrations. We can apply the filter using the following query:

{
-  courses {    
+  courses(hasCourseRegistrations: {column: CONFIRMATION_DATE, operator: EQ, value: null}) {
    id
    description
    courseRegistrations {
      id
      confirmation_date
    }
  }
}

NOTE: All basic SQL operators are supported but the most common ones are EQ and NEQ.

Authorization

Required roles: admin.

Registering for a course

A user can be registered for a course using the registerForCourse mutation. This requires a single argument for the course.

mutation{
  registerForCourse(course: 2){
    id,
    course {
      id
      subject
    }
  }
}

By default, the user will be loaded from the authentication context, but this can be overriden in two ways. First, we can explicitly define the user to register:

mutation{
-   registerForCourse(course: 42){
+   registerForCourse(course: 42, user: 13){
        id,
    course {
      id
      subject
    }
    }
}

Or, we can define a first name and a surname which will create a user object for you. This way, users don't have to be registered to register for a course

mutation{
-   registerForCourse(course: 42){
+   registerForCourse(course: 42, first_name: "Steve", surname: "Wozniak"){
        id,
    course {
      id
      subject
    }
    }
}

NOTE: Users can only register for a course once.

Return value

The mutation returns the new course registration.

{
  "data": {
    "registerForCourse": {
      "id": "16",
      "course": {
        "id": "2",
        "subject": "Praktisch leidinggeven"
      }
    }
  }
}

Authorization

none.

Required parameters

In order for a user to register for a course, the following parameters have to be provided. Failing to provide any of the following parameters will result in an error being thrown.

View parameters

course: String

Optional parameters

In addition to the required parameters, the API accepts the following optional parameters.

View parameters

first_name: String
surname: String
user: ID
study_registration: ID
intermediary_user: ID
registration_number: String
payment_number: String
price: Price
role: String define the role this user has within this registration. Will default to "student", if left out.

Creating a new course

As an administrator you are able to create new courses that can be fetched by client applications. To provide this functionality the createCourse mutation is created.

mutation {
  createCourse(subject: "subject", description: "description")  {
    id
    subject
    description
  }
}

Return value

The mutation returns the newly created course.

{
  "data": {
    "createCourse": {
      "id": "15",
      "subject": "subject",
      "description": "description"
    }
  }
}

Authorization

Required roles: admin.

Required parameters

The createCourse mutation only requires the following two parameters, although we advise to fill in as much of the information as possible. This will help users of your application to make an informed decision about a course.

View parameters description: String!
subject: String!

Optional parameters

In addition to the required parameters, the API accepts the following optional parameters.

View parameters parent_course: ID
code: String
price: Price
period: String
min_participants: Int
max_participants: Int
comment: String
courseType: ID
location: Int
course_number: String
project_code: String
head_teacher: ID
content: String
learning_goals: String
template: Boolean
certificate_validity_months: Int
certificate_validity_start: DateTime
registration_end_date: DateTime
registration_start_date: DateTime studies: [ID]

Updating and existing course

Administrators should be able to update a course for a number of different reasons. An example of such a reason is to update the price for a course. We offer this functionality through the updateCourse mutation.

mutation {
  updateCourse(id: 1, price: "80.00") {
    id
    description
    price
  }
}

Return value

Assuming that the request is successfull, and the user has provided his or her bearer token, the response should look something like this.

{
  "data": {
    "updateCourse": {
      "id": "1",
      "description": "Computer Science",
      "price": "80.00"
    }
  }
}

Authorization

Required roles: admin.

Required parameters

This mutation has a single required parameter, the id of the course to be mutated:

View parameters id: ID!

Optional parameters

In addition to the required parameters, the API accepts the following optional parameters.

View parameters parent_course: ID
price: Price
period: String
comment: String
subject: String
description: String
min_participants: Int
max_participants: Int
course_number: String
project_code: String
head_teacher: ID
courseType: ID
content: String
learning_goals: String
template: Boolean
archived: Boolean
certificate_validity_months: Int
certificate_validity_start: DateTime
cancellation_time: DateTime
registration_start_date: DateTime
registration_end_date: DateTime

Deleting an existing course

Sometimes a course is no longer needed. In these cases we enable administrators to delete courses using one of two mutations: deleteCourse & forceDeleteCourse

Difference between the two mutations

We offer two ways of deleting courses. The deleteCourse mutation uses a principle called soft deletion. This means that the soft deleted courses are no longer included in queries but they still remain in the database. This makes it possible to restore courses that are deleted by accident. This is done using the restoreCourse mutation.

In contrast to this we also offer a solution for cases where a course should actually be deleted from the database. For those use cases we offer the forceDeleteCourse mutation.

Soft deleting a course

As described in the previous chapter this mutation makes use of a concept called soft deletion. This does not actually deletes the course from the database but simply makes sure that soft deleted courses are excluded from all queries.

mutation {
  deleteCourse(id: 2) {
    id
    subject
    deleted_at
  }
}

Return value

The (to be) deleted course will be returned.

{
  "data": {
    "deleteCourse": {
      "id": "2",
      "subject": "Praktisch leidinggeven",
      "deleted_at": "2021-01-12 12:00:00"
    }
  }
}

Authorization

Required roles: admin.

Required parameters

This mutation has a single required parameter, the id of the course to be soft deleted:

View parameters id: ID!

Optional parameters

This mutation has no optional parameters

Restore course

Sometimes a course has been deleted by accident or you might need to restore it for another reason. When it was deleted using the soft delete concept this is done using the restoreCourse mutation.

mutation {
  restoreCourse(id: 2) {
    id
    subject
    deleted_at
  }
}

Return value

Assuming that the request is successfull, and the user has provided his or her bearer token, the response should look something like this.

{
  "data": {
    "restoreCourse": {
      "id": "2",
      "subject": "Praktisch leidinggeven",
      "deleted_at": null
    }
  }
}

Authorization

Required roles: admin.

Required parameters

This mutation has a single required parameter, the id of the course to be restored:

View parameters id: ID!

Optional parameters

This mutation has no optional parameters

Force delete course

When simply excluding a course from queries isn't enough, this mutation can be used to completely remove the course from the database.

mutation {
  forceDeleteCourse(id: 7) {
    id
    subject
    deleted_at
  }
}

NOTE: You cannnot force delete courses that has registered users, since this would break existing relations. Opt to use softw delete, or remove the relations first.

Return value

Assuming that the request is successfull, and the user has provided his or her bearer token, the response should look something like this.

{
  "data": {
    "forceDeleteCourse": {
      "id": "7",
      "subject": "Asbest herkennen",
      "deleted_at": null
    }
  }
}

NOTE: Please note that hard deleting does not edit the deleted_at field. This field is only updated when soft deleting courses.

Authorization

Required roles: admin.

Required parameters

This mutation has a single required parameter, the id of the course to be mutated:

View parameters id: ID!

Optional parameters

This mutation has no optional parameters

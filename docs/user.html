Users

The user is at the core of the Promptus API. On this page, we will discuss the queries, mutations, and fields related to users. Detailed information of the User entity can be found in the API reference.

Roles

Primarily, users are people that want to register for courses. We call these users 'students'.
Of course, their teachers can also be users. As can the employees working in the back-office of the teaching institute company. Every person who has registered in your Promptus system is in fact a user. To identify who does what, we can assign users with one or more roles. Each role has permissions, which can be required for certain queries and mutations. Learn more about this on the Authenticate User page. We identify the following roles:

  • Student
  • Teacher
  • Actor
  • Account manager
  • Employee
  • Admin

Get active user

If in any case you want to return the user that is currently active (logged in), you can do so using the me query:

{
  me {
    id
    fullName
  }
}

Return value

This returns the active user, based on provided bearer token.

{
  "data": {
    "me": {
      "id": 1,
      "first_name": "John"
    }
  }
}

Authorization

Requires user to provide their access (bearer) token.

Get user by id

You can query specific users by their id:

{
  user (1) {
    id
    fullName
  }
}

Return value

This returns the user with the corresponding id.

{
  "data": {
    "user": {
      "id": 1,
      "first_name": "John"
    }
  }
}

Authorization

None.

Search users by search_code

You can query specific users by search code:

{
  searchUsers ("John") {
    id
    fullName
  }
}

Return value

This returns the active user, based on provided bearer token.

{
  "data": {
    "searchUsers": {
      "id": 1,
      "first_name": "John"
    }
  }
}

Authorization

None.

Get a list of all users with a specific role

You can get a list of all users with a specific role:

{
  usersByRole(role: "students") {
    first_name
    is_student
  }
}

Note: Provided role must be one of the following: account_managers, actors, administrators, employees, students, teachers.

Return value

This returns the active user, based on provided bearer token.

{
  "data": {
    "usersByRole": [
      {
        "first_name": "John",
        "is_student": true
      },
      {
        "first_name": "Jim",
        "is_student": true
      },
      {
        "first_name": "David",
        "is_student": true
      },
      {
        "first_name": "Edward",
        "is_student": true
      }
    ]
  }
}

Authorization

Required roles: admin.

Registering a new user

To add a user, we will have to register him or her. As discussed in the getting started page, the only way to add data is by using mutations. The mutation to register a user is called: registerUser:

mutation {
  registerUser(input: {first_name: "John", surname: "Doe" email: "johndoe@example.com", password: "secret"}) {
    id
    first_name
  }
}

Note: It is up to you (the client application) to implement features like input validation.

Return value

The mutation returns the newly created user.

{
  "data": {
    "registerUser": {
      "id": 1,
      "first_name": "John",
      "teacher": {
        "id": null // John was not assigned the role of teacher upon registration. 
      },
      "student": {
        "id": 1 // John will automatically be assigned the role of student, since no other role was given. 
      }
    }
  }
}

Authorization

none.

Updating an existing user

There are various reasons why users might want to update their own information. An example of this can be that the user wants to add extra contact information. It's also possible for admins to change information of other users. To account for these situations we provide the updateProfileInformation mutation:

mutation {
  updateProfileInformation(gsm_number: "0612345678") {
    first_name
    gsm_number
  }
}

Return value

Assuming that the request is successfull, and the user has provided his or her bearer token, the response should look something like this.

{
  "data": {
    "updateProfileInformation": {
      "first_name": "John",
      "gsm_number": "0612345678",
    }
  }
}

Authorization

Required roles: admin (when updating other users).
Users with other roles can update their own information if they have provided their user (bearer) token.

Note: When an admin wants to update another user, that user has to be identified by providing the id field. When leaving out the id field, this mutation assumes you are updating the current user, which is determined by the bearer token.

There are no required fields for this mutation.
